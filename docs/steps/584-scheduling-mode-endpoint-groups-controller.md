# 584 — Scheduling Mode Endpoint on GroupsController

## Phase

Phase 14 — API Layer (Self-Service Scheduling)

## Purpose

Exposes the `ChangeSchedulingModeCommand` (implemented in task 4.1) via a REST endpoint on the existing `GroupsController`. This allows admins/owners to toggle a group between `AutoGenerated` and `SelfService` scheduling modes through the API.

## What was built

| File | Description |
|------|-------------|
| `apps/api/Jobuler.Api/Controllers/GroupsController.cs` | Added `PUT spaces/{spaceId}/groups/{groupId}/scheduling-mode` endpoint with `ChangeSchedulingModeRequest` DTO. Added imports for `Jobuler.Application.Scheduling.Commands` and `Jobuler.Domain.Groups`. |

## Key decisions

1. **PUT verb**: Used PUT since the operation is idempotent — setting the same mode twice is a no-op (handled by the command handler).
2. **Defense-in-depth permission check**: The controller checks `Permissions.SchedulePublish` before dispatching, even though the command handler also validates permission. This follows the architecture rule that every write controller action must call `RequirePermissionAsync`.
3. **Tenant isolation**: The `spaceId` route parameter is passed to the command, which filters by `space_id` in the query. Combined with RLS, this ensures tenant isolation.
4. **No new controller**: Added to the existing `GroupsController` as specified, keeping the endpoint co-located with other group management operations.

## How it connects

- Dispatches `ChangeSchedulingModeCommand` (task 4.1) via MediatR
- The command handler validates no active `Pending`/`Approved` shift requests exist before allowing mode change (Requirement 1.5)
- Self-service groups bypass the solver entirely (Requirement 1.2)
- Auto-generated groups reject shift request submissions (Requirement 1.3)

## How to run / verify

```bash
dotnet build apps/api/Jobuler.Api
```

Example API call:
```
PUT /spaces/{spaceId}/groups/{groupId}/scheduling-mode
Authorization: Bearer <token>
Content-Type: application/json

{ "mode": "SelfService" }
```

Returns `204 No Content` on success, `400` if active requests exist, `403` if insufficient permission.

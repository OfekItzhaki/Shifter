# Step 572 — ChangeSchedulingMode Command and Solver Guard

## Phase

Self-Service Scheduling — Application Layer (Task 4.1)

## Purpose

Implements the command to change a group's scheduling mode between `AutoGenerated` and `SelfService`, with safety checks to prevent mode changes while active shift requests exist. Also guards the existing `TriggerSolverCommand` to short-circuit for SelfService groups, ensuring the solver never runs for groups that manage shifts through the request-based flow.

## What was built

| File | Description |
|------|-------------|
| `apps/api/Jobuler.Application/Scheduling/Commands/ChangeSchedulingModeCommand.cs` | Command record, FluentValidation validator, and MediatR handler for changing scheduling mode |
| `apps/api/Jobuler.Application/Scheduling/Commands/TriggerSolverCommand.cs` | Added SelfService guard that throws `InvalidOperationException` when a solver run is triggered for a SelfService group |
| `apps/api/Jobuler.Tests/Scheduling/ChangeSchedulingModeCommandTests.cs` | Unit tests covering: success cases, no-op for same mode, rejection with active requests, permission denial, inactive group, group not found |
| `apps/api/Jobuler.Tests/Scheduling/TriggerSolverSelfServiceGuardTests.cs` | Unit tests verifying the solver guard rejects SelfService groups and allows AutoGenerated groups |

## Key decisions

1. **Permission check uses `SchedulePublish`** — Changing scheduling mode is a high-impact operation equivalent to publishing schedule decisions, so it requires the same permission level as publishing.
2. **Active request check is broad** — Checks for ANY Pending or Approved request in the group (not scoped to a specific cycle), ensuring no data is orphaned during mode transitions.
3. **No-op for same mode** — If the group is already in the target mode, the handler returns immediately without touching the database.
4. **Guard placement in TriggerSolverCommand** — The SelfService guard is placed immediately after the existing `EnsureActive()` call, before any solver-related work begins.

## How it connects

- **Requirement 1.2**: SelfService groups cannot trigger solver runs (enforced in `TriggerSolverCommand`)
- **Requirement 1.3**: AutoGenerated groups cannot have self-service requests (enforced at request submission time in future task 7.3)
- **Requirement 1.5**: Mode change blocked by active Pending/Approved requests
- **Requirement 1.6**: In-progress solver runs are allowed to complete (the guard only prevents NEW triggers)

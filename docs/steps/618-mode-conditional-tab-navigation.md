# 618 — Mode-Conditional Tab Navigation

## Phase
Self-Service Scheduling UI — Tab Integration

## Purpose
Extends the GroupDetailPage to conditionally render tabs based on the group's `schedulingMode` field. Self-service groups show member/admin self-service tabs while auto-generated groups continue showing the existing solver-related tabs.

## What was built

| File | Change |
|------|--------|
| `apps/web/app/groups/[groupId]/types.ts` | Extended `ActiveTab` type with 7 self-service tab values; added `AUTO_GENERATED_TABS`, `SELF_SERVICE_MEMBER_TABS`, `SELF_SERVICE_ADMIN_TABS` arrays |
| `apps/web/lib/api/groups.ts` | Added `schedulingMode?: "AutoGenerated" \| "SelfService"` to `GroupWithMemberCountDto` |
| `apps/web/app/groups/[groupId]/page.tsx` | Added lazy imports for all 7 self-service tab components; extended `getTabLabels` with self-service i18n keys; mode-conditional `visibleTabs` logic; default tab based on scheduling mode; self-service tab content rendering; extended members/tasks loading for self-service tabs |

## Key decisions

1. **Tab arrays as constants in types.ts** — keeps the tab configuration declarative and testable, separate from the rendering logic.
2. **Default tab set on group load** — when the group entity is fetched, `setActiveTab` is called with `"available-slots"` for self-service or `"schedule"` for auto-generated, ensuring the correct initial view.
3. **Lazy-loading pattern** — all self-service tabs use `React.lazy()` + dynamic imports, consistent with existing tabs (AlertsTab, StatsTab, etc.).
4. **Members/tasks loading extended** — the existing data-fetching effects now also trigger for self-service tabs that need those data (swaps, admin-overrides need members; shift-templates needs tasks).
5. **`schedulingMode` as optional field** — added as optional to `GroupWithMemberCountDto` to maintain backward compatibility with existing groups that don't have the field set.

## How it connects

- Depends on the backend `SchedulingMode` enum already present on the `Group` entity and returned in the groups API response.
- Depends on self-service tab components already created in previous tasks (SlotBrowserTab, MyShiftsTab, WaitlistTab, SwapsTab, ShiftTemplatesTab, SelfServiceConfigTab, AdminOverridesTab).
- Depends on i18n keys under `selfService.tabs.*` namespace already added in task 3.1.
- The `GroupDetailPage` is the central integration point that wires all self-service UI together.

## How to run / verify

1. Load a group with `schedulingMode: "SelfService"` — should see self-service tabs (available-slots, my-shifts, etc.) and default to "available-slots".
2. Load a group with `schedulingMode: "AutoGenerated"` (or undefined) — should see existing tabs and default to "schedule".
3. In self-service mode as non-admin: should NOT see shift-templates, self-service-config, admin-overrides tabs.
4. In self-service mode as admin: should see all self-service tabs including admin-only ones.
5. TypeScript compilation passes with no errors.

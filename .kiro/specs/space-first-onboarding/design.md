# Technical Design Document

## Overview

This design implements the space-first onboarding flow, space invite system, linked groups (parent-child), space switching, and migration for existing users. The system already has `Space`, `SpaceMembership`, and `SpacePermissionGrant` entities with full CQRS handlers. The primary changes are: (1) adding an invite code to Space, (2) adding `parent_group_id` to Group, (3) building the onboarding wizard UI, (4) adding space-switching UI, and (5) implementing the migration service.

## 1. Database Schema Changes

### 1.1 Space Table — Add Invite Code

```sql
ALTER TABLE spaces ADD COLUMN invite_code VARCHAR(8) UNIQUE;
```

The `invite_code` is generated on Space creation and can be regenerated by the owner.

### 1.2 Group Table — Add Parent Group

```sql
ALTER TABLE groups ADD COLUMN parent_group_id UUID REFERENCES groups(id) ON DELETE SET NULL;
```

Constraint: single-level hierarchy only (a group with a parent cannot itself be a parent).

### 1.3 User Migration Tracking

```sql
CREATE TABLE user_space_migrations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id),
    space_id UUID NOT NULL REFERENCES spaces(id),
    migrated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    groups_migrated INT NOT NULL DEFAULT 0,
    UNIQUE(user_id)
);
```

## 2. Domain Layer Changes

### 2.1 Space Entity Updates

```csharp
// Add to Space.cs
public string? InviteCode { get; private set; }

// In Create method:
InviteCode = GenerateInviteCode();

public string RegenerateInviteCode()
{
    InviteCode = GenerateInviteCode();
    Touch();
    return InviteCode;
}

private static string GenerateInviteCode() =>
    Guid.NewGuid().ToString("N")[..8].ToUpperInvariant();
```

### 2.2 Group Entity Updates

```csharp
// Add to Group.cs
public Guid? ParentGroupId { get; private set; }

public void SetParentGroup(Guid? parentGroupId)
{
    ParentGroupId = parentGroupId;
    Touch();
}

public void UnlinkFromParent()
{
    ParentGroupId = null;
    Touch();
}
```

### 2.3 New Entity: UserSpaceMigration

```csharp
public class UserSpaceMigration : Entity
{
    public Guid UserId { get; private set; }
    public Guid SpaceId { get; private set; }
    public DateTime MigratedAt { get; private set; } = DateTime.UtcNow;
    public int GroupsMigrated { get; private set; }

    public static UserSpaceMigration Create(Guid userId, Guid spaceId, int groupsMigrated) =>
        new() { UserId = userId, SpaceId = spaceId, GroupsMigrated = groupsMigrated };
}
```

## 3. API Endpoints

### 3.1 New Endpoints

| Method | Route | Description | Auth |
|--------|-------|-------------|------|
| POST | `/spaces/{spaceId}/invite-code/regenerate` | Regenerate invite code (owner only) | Owner |
| POST | `/spaces/join` | Join a space via invite code | Authenticated |
| GET | `/spaces/{spaceId}/members` | List space members | Member |
| PUT | `/spaces/{spaceId}` | Update space settings | Owner |
| POST | `/spaces/{spaceId}/groups/{groupId}/link-parent` | Set parent group | Owner |
| DELETE | `/spaces/{spaceId}/groups/{groupId}/link-parent` | Remove parent link | Owner |
| POST | `/spaces/migrate` | Trigger migration for current user | Authenticated |

### 3.2 Modified Endpoints

| Method | Route | Change |
|--------|-------|--------|
| GET | `/spaces` | Already exists — returns user's spaces |
| POST | `/spaces` | Already exists — now also generates invite code |
| GET | `/spaces/{spaceId}` | Add `inviteCode`, `memberCount`, `groupCount` to response |

## 4. Application Layer — New Commands/Queries

### 4.1 Commands

- `JoinSpaceByInviteCodeCommand(string InviteCode, Guid UserId)` → `JoinSpaceResponse`
- `RegenerateSpaceInviteCodeCommand(Guid SpaceId, Guid RequestingUserId)` → `string`
- `UpdateSpaceCommand(Guid SpaceId, string Name, string? Description, string Locale, Guid RequestingUserId)`
- `LinkParentGroupCommand(Guid SpaceId, Guid ChildGroupId, Guid ParentGroupId, Guid RequestingUserId)`
- `UnlinkParentGroupCommand(Guid SpaceId, Guid ChildGroupId, Guid RequestingUserId)`
- `MigrateUserSpaceCommand(Guid UserId)` → `Guid` (returns new space ID)

### 4.2 Queries

- `GetSpaceMembersQuery(Guid SpaceId)` → `List<SpaceMemberDto>`
- `GetSpaceDetailQuery(Guid SpaceId, Guid RequestingUserId)` → `SpaceDetailDto`

### 4.3 Validators

- `JoinSpaceByInviteCodeValidator` — code must be exactly 8 alphanumeric chars
- `UpdateSpaceValidator` — name 2–100 chars, description 0–500 chars
- `LinkParentGroupValidator` — both groups in same space, no circular refs, single-level

## 8. Migration Service Design

The migration runs on login when:
1. User has groups (CreatedByUserId = current user)
2. User has NO SpaceMembership records
3. No `user_space_migrations` record exists for this user

Migration steps (in a transaction):
1. Create Space named `"{displayName}'s Space"` (or "My Space" if name is blank)
2. Set user as owner, create SpaceMembership
3. Update all user's groups where `space_id` is null → set to new space ID
4. Record migration in `user_space_migrations`
5. Return new space ID to frontend

The migration is triggered from the frontend during the redirect logic check.

## 9. Parent-Child Group Linking

### Validation Rules:
1. Both groups must belong to the same space
2. Parent group cannot itself have a parent (single-level only)
3. Child group cannot already be a parent of other groups

### Cascading Schedule:
When a parent group's schedule is published, the solver payload for child groups includes:
```json
{
  "parent_schedule": {
    "assignments": [...],
    "version_id": "..."
  }
}
```

The solver uses this to avoid conflicts between parent and child schedules.

## 10. Redirect Logic (Middleware)

Frontend middleware in `apps/web/middleware.ts`:

```typescript
// On every authenticated page load:
// 1. Check if user has spaces (from API or cached)
// 2. If no spaces → redirect to /onboarding
// 3. If has spaces but stored spaceId is invalid → pick first space
// 4. If on /onboarding but has spaces → redirect to /home
```

## 11. Permission Model

No new permission keys needed. Existing permissions cover all operations:
- `space.view` — granted to all members on join
- `space.admin_mode` — required for group linking, space settings
- `permissions.manage` — required for granting/revoking permissions

Space owner implicitly holds all permissions (enforced in `PermissionService`).

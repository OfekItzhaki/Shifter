# Requirements Document

## Introduction

This feature builds the frontend UI for the self-service scheduling system in the Shifter web application. The backend is already fully implemented (shift templates, slot generation, shift requests, waitlist, swaps, admin overrides). This spec covers the user-facing Next.js pages and components that integrate with those existing API endpoints.

The UI introduces a **scheduling mode choice** during group creation (permanent and irreversible), mode-specific tabs on the group detail page, and all member/admin interactions for the self-service flow: browsing slots, requesting/cancelling shifts, managing waitlists, proposing swaps, and admin override actions.

## Glossary

- **Scheduling_Mode_Selector**: The UI component in the group creation wizard that presents the two scheduling modes as visual cards
- **Group_Creation_Wizard**: The multi-step modal flow for creating a new group, extended to include scheduling mode selection
- **Shift_Template_Manager**: The admin UI for creating, editing, and deleting recurring shift templates
- **Slot_Browser**: The member-facing UI that displays available shift slots for the current cycle
- **Request_Panel**: The UI component that allows members to request or cancel shifts
- **Waitlist_Panel**: The UI component showing waitlist status and allowing join/leave/accept actions
- **Swap_Panel**: The UI component for proposing, accepting, declining, and cancelling shift swaps
- **Admin_Override_Panel**: The admin UI for force-assigning or removing members from shifts
- **Self_Service_Config_Panel**: The admin UI for configuring min/max shifts, request windows, and cancellation cutoff
- **Mode_Warning_Dialog**: A confirmation dialog warning that scheduling mode choice is permanent
- **Group_Detail_Page**: The existing `/groups/[groupId]` page that renders tabs based on group context
- **API_Client**: The existing `apiClient` module used for all HTTP requests to the backend

## Requirements

### Requirement 1: Scheduling Mode Selection in Group Creation

**User Story:** As a space admin, I want to choose between automatic scheduling and self-service scheduling when creating a group, so that the group is configured with the correct scheduling approach from the start.

#### Acceptance Criteria

1. WHEN the Group_Creation_Wizard opens, THE Scheduling_Mode_Selector SHALL present two visual cards: "סידור אוטומטי" (Automatic Scheduling) and "בחירת משמרות" (Shift Picking)
2. THE Scheduling_Mode_Selector SHALL display a clear description on each card explaining what the mode means: automatic mode uses the solver to generate schedules, self-service mode lets members pick their own shifts
3. THE Scheduling_Mode_Selector SHALL display a prominent warning message indicating that the scheduling mode choice is permanent and cannot be changed after group creation
4. WHEN the user selects a scheduling mode card, THE Group_Creation_Wizard SHALL visually highlight the selected card with a distinct border and background color
5. WHEN the user has not selected a scheduling mode, THE Group_Creation_Wizard SHALL disable the continue/submit button
6. WHEN the user selects "סידור אוטומטי", THE Group_Creation_Wizard SHALL show the existing template picker (army-base, restaurant, hospital, security, custom) and the "stay over" (home-leave) toggle option
7. WHEN the user selects "בחירת משמרות", THE Group_Creation_Wizard SHALL show self-service-specific template options for shift patterns
8. WHEN the user clicks continue after selecting a mode and template, THE Mode_Warning_Dialog SHALL appear with a final confirmation that the choice is irreversible, requiring explicit acknowledgment before proceeding
9. WHEN the user confirms the mode selection, THE Group_Creation_Wizard SHALL call the group creation API with the selected `schedulingMode` value and template configuration
10. IF the group creation API returns an error, THEN THE Group_Creation_Wizard SHALL display the error message and allow the user to retry without losing their selections

### Requirement 2: Mode-Conditional Group Page Tabs

**User Story:** As a group member or admin, I want the group page to show relevant tabs based on the scheduling mode, so that I only see features applicable to my group's scheduling approach.

#### Acceptance Criteria

1. WHEN the Group_Detail_Page loads a group with `schedulingMode` equal to `AutoGenerated`, THE Group_Detail_Page SHALL display the existing tabs: schedule, live-status, members, qualifications, roles, alerts, messages, tasks, constraints, stats, settings
2. WHEN the Group_Detail_Page loads a group with `schedulingMode` equal to `SelfService`, THE Group_Detail_Page SHALL display tabs: available-slots, my-shifts, waitlist, swaps, members, qualifications, roles, alerts, messages, settings, and for admins additionally: shift-templates, self-service-config, admin-overrides
3. WHEN a non-admin member views a self-service group, THE Group_Detail_Page SHALL hide admin-only tabs (shift-templates, self-service-config, admin-overrides)
4. THE Group_Detail_Page SHALL default to the "available-slots" tab for self-service groups and the "schedule" tab for auto-generated groups
5. WHEN the scheduling mode is `SelfService`, THE Group_Detail_Page SHALL NOT display the solver-related tabs (constraints, tasks, live-status, stats)

### Requirement 3: Shift Template Management UI (Admin)

**User Story:** As a group admin, I want to manage shift templates through a visual interface, so that I can define recurring shift patterns without using the API directly.

#### Acceptance Criteria

1. WHEN an admin navigates to the shift-templates tab, THE Shift_Template_Manager SHALL display a list of all shift templates for the group, showing day of week, start time, end time, required headcount, and associated task name
2. THE Shift_Template_Manager SHALL provide a form to create a new shift template with fields: day of week (dropdown, Sunday through Saturday), start time (time picker), end time (time picker), required headcount (number input, 1-999), and associated group task (dropdown)
3. WHEN an admin submits a new template with start time equal to or after end time, THE Shift_Template_Manager SHALL display a validation error without calling the API
4. WHEN an admin submits a valid template, THE Shift_Template_Manager SHALL call the POST shift-templates API endpoint and add the new template to the displayed list on success
5. THE Shift_Template_Manager SHALL allow editing an existing template's fields and call the PUT shift-templates API endpoint on save
6. THE Shift_Template_Manager SHALL allow deleting a template with a confirmation dialog, calling the DELETE shift-templates API endpoint on confirm
7. IF the API returns an error during create, update, or delete, THEN THE Shift_Template_Manager SHALL display the error message from the response to the admin

### Requirement 4: Self-Service Configuration UI (Admin)

**User Story:** As a group admin, I want to configure self-service scheduling parameters through a settings panel, so that I can control shift limits and request windows for my group.

#### Acceptance Criteria

1. WHEN an admin navigates to the self-service-config tab, THE Self_Service_Config_Panel SHALL display the current configuration values: min shifts per cycle, max shifts per cycle, request window open offset (hours), request window close offset (hours), cancellation cutoff (hours), waitlist offer duration (minutes), and cycle duration (days)
2. THE Self_Service_Config_Panel SHALL provide editable inputs for each configuration field with appropriate input types (number inputs with min/max constraints)
3. WHEN an admin enters a min shifts value greater than max shifts, THE Self_Service_Config_Panel SHALL display a client-side validation error indicating min must be less than or equal to max
4. WHEN an admin enters values outside the valid ranges (min: 0-100, max: 1-100, offsets: 1-720 hours), THE Self_Service_Config_Panel SHALL display a client-side validation error indicating the valid range
5. WHEN an admin submits valid configuration, THE Self_Service_Config_Panel SHALL call the PUT self-service-config API endpoint and display a success confirmation
6. IF the API returns an error, THEN THE Self_Service_Config_Panel SHALL display the error message and preserve the user's input for correction

### Requirement 5: Slot Browser UI (Member)

**User Story:** As a group member, I want to browse available shift slots for the current cycle, so that I can choose shifts that fit my schedule.

#### Acceptance Criteria

1. WHEN a member navigates to the available-slots tab, THE Slot_Browser SHALL call the GET available-slots API endpoint and display all slots with remaining capacity, sorted by date ascending then start time ascending
2. THE Slot_Browser SHALL display for each slot: date, day of week name (in Hebrew), start time, end time, task name, and a capacity indicator showing current fill count versus total capacity (e.g., "2/5")
3. WHEN the request window is closed, THE Slot_Browser SHALL display a banner indicating that requests are not currently accepted and show the next window opening date/time
4. WHILE the request window is open, THE Slot_Browser SHALL display a "Request" button on each available slot
5. WHEN a member clicks the "Request" button, THE Slot_Browser SHALL call the POST shift-requests API endpoint and update the slot's capacity indicator on success
6. IF the shift request is rejected due to full capacity, THEN THE Slot_Browser SHALL display the rejection reason and show up to 5 alternative slots for the same day if provided by the API
7. IF the shift request is rejected due to max shifts reached, THEN THE Slot_Browser SHALL display a message indicating the member has reached their maximum shift count for the cycle
8. WHEN a slot is at full capacity, THE Slot_Browser SHALL display a "Join Waitlist" button instead of the "Request" button
9. THE Slot_Browser SHALL provide a date filter allowing members to view slots for specific days within the cycle
10. THE Slot_Browser SHALL visually distinguish between slots with high availability (more than 50% capacity remaining) and slots nearly full (50% or less remaining)

### Requirement 6: My Shifts View (Member)

**User Story:** As a group member, I want to view my approved shifts and pending requests, so that I can see my complete schedule at a glance.

#### Acceptance Criteria

1. WHEN a member navigates to the my-shifts tab, THE Request_Panel SHALL display all of the member's shift requests for the current cycle, grouped by status: approved, pending, and cancelled
2. THE Request_Panel SHALL display for each shift: date, day of week (Hebrew), start time, end time, task name, and current status with a color-coded badge
3. WHEN a member has an approved shift that is within the cancellation window, THE Request_Panel SHALL display a "Cancel" button next to that shift
4. WHEN a member clicks "Cancel", THE Request_Panel SHALL show a dialog requiring a cancellation reason (1-500 characters) before calling the POST cancel-request API endpoint
5. IF the cancellation is rejected because the cutoff has passed, THEN THE Request_Panel SHALL display an error message indicating the cancellation window has closed
6. THE Request_Panel SHALL display the member's current shift count relative to the configured min/max (e.g., "3 / 5 shifts this cycle")
7. WHEN the member's shift count is below the minimum, THE Request_Panel SHALL display a warning indicating they are under-scheduled

### Requirement 7: Waitlist UI (Member)

**User Story:** As a group member, I want to manage my waitlist entries and respond to slot offers, so that I can get shifts that become available after cancellations.

#### Acceptance Criteria

1. WHEN a member navigates to the waitlist tab, THE Waitlist_Panel SHALL display all of the member's active waitlist entries showing: slot date, time, task name, current position in queue, and status
2. WHEN a member has a pending offer (status "Offered"), THE Waitlist_Panel SHALL display the offer prominently with the acceptance deadline countdown and "Accept" / "Decline" buttons
3. WHEN a member clicks "Accept" on an offer, THE Waitlist_Panel SHALL call the POST accept-offer API endpoint and move the shift to the my-shifts view on success
4. IF the acceptance is rejected due to max shifts, THEN THE Waitlist_Panel SHALL display a message indicating the member has reached their limit and the offer has been passed to the next person
5. WHEN a member clicks "Decline" or the offer expires, THE Waitlist_Panel SHALL update the entry status accordingly
6. THE Waitlist_Panel SHALL allow members to leave a waitlist by clicking a "Leave" button, with a confirmation prompt
7. WHEN a member clicks "Join Waitlist" from the Slot_Browser, THE Waitlist_Panel SHALL call the POST join-waitlist API endpoint and display the new entry with the assigned position

### Requirement 8: Shift Swap UI (Member)

**User Story:** As a group member, I want to propose and manage shift swaps with other members, so that I can adjust my schedule collaboratively.

#### Acceptance Criteria

1. WHEN a member navigates to the swaps tab, THE Swap_Panel SHALL display all of the member's swap requests (both initiated and received) with status, counterpart name, offered shift details, and requested shift details
2. THE Swap_Panel SHALL provide a "Propose Swap" flow where the member selects one of their approved shifts to offer and one of another member's approved shifts to request
3. WHEN a member submits a swap proposal, THE Swap_Panel SHALL call the POST propose-swap API endpoint and display the new pending swap on success
4. IF the swap proposal is rejected (duplicate pending swap, invalid ownership), THEN THE Swap_Panel SHALL display the specific error message from the API
5. WHEN a member receives a swap proposal, THE Swap_Panel SHALL display "Accept" and "Decline" buttons on the incoming request
6. WHEN a member accepts a swap, THE Swap_Panel SHALL call the POST accept-swap API endpoint and update both shifts in the my-shifts view on success
7. IF the swap acceptance is rejected due to a time conflict, THEN THE Swap_Panel SHALL display the conflict details indicating which shifts overlap
8. WHEN a member declines a swap, THE Swap_Panel SHALL call the POST decline-swap API endpoint and update the request status
9. WHEN a member cancels their own pending swap, THE Swap_Panel SHALL call the POST cancel-swap API endpoint and remove the request from the active list
10. THE Swap_Panel SHALL display a countdown timer for pending swaps showing time remaining before the 72-hour expiry

### Requirement 9: Admin Override UI

**User Story:** As a group admin, I want to manually assign or remove members from shifts through the UI, so that I can handle exceptions without using the API directly.

#### Acceptance Criteria

1. WHEN an admin navigates to the admin-overrides tab, THE Admin_Override_Panel SHALL display the current cycle's shift slots with their assigned members and capacity status
2. THE Admin_Override_Panel SHALL provide an "Assign Member" action on each slot, opening a member picker dropdown filtered to group members not already assigned to that slot
3. WHEN an admin selects a member and confirms assignment, THE Admin_Override_Panel SHALL call the POST admin-assign API endpoint and update the slot display on success
4. THE Admin_Override_Panel SHALL provide a "Remove" action next to each assigned member on a slot
5. WHEN an admin clicks "Remove" and confirms, THE Admin_Override_Panel SHALL call the POST admin-remove API endpoint and update the slot display on success
6. IF the admin does not hold the `SchedulePublish` permission, THEN THE Admin_Override_Panel SHALL display a message indicating insufficient permissions and disable all override actions
7. IF the API returns an error (member not in group, already assigned, not assigned), THEN THE Admin_Override_Panel SHALL display the specific error message from the response

### Requirement 10: RTL and Hebrew Localization

**User Story:** As a Hebrew-speaking user, I want all self-service scheduling UI to be properly localized and RTL-aligned, so that the interface is natural and readable in my language.

#### Acceptance Criteria

1. THE system SHALL render all self-service scheduling components in RTL layout direction consistent with the existing application RTL support
2. THE system SHALL use Hebrew text for all labels, buttons, messages, error texts, and descriptions in the self-service scheduling UI, sourced from the existing next-intl message files
3. THE system SHALL display dates in Hebrew locale format and day-of-week names in Hebrew (ראשון, שני, שלישי, רביעי, חמישי, שישי, שבת)
4. THE system SHALL display time values in 24-hour format consistent with the existing application time display conventions
5. WHEN an API error message is received, THE system SHALL display a localized Hebrew error message mapped from the error code, falling back to a generic Hebrew error message if no specific mapping exists

### Requirement 11: API Client Integration

**User Story:** As a developer, I want the frontend to use typed API client functions for all self-service endpoints, so that the integration is type-safe and consistent with the existing codebase patterns.

#### Acceptance Criteria

1. THE system SHALL provide typed API client functions in a dedicated module (e.g., `lib/api/selfService.ts`) for all self-service scheduling endpoints: shift-templates CRUD, self-service-config GET/PUT, available-slots GET, shift-requests POST/cancel/list, waitlist join/leave/accept/list, swaps propose/accept/decline/cancel/list, admin assign/remove
2. THE API_Client functions SHALL use the existing `apiClient` instance and follow the same patterns as other API modules (async functions returning typed responses)
3. THE API_Client functions SHALL include the space ID and group ID path parameters consistent with the existing URL patterns (`/spaces/{spaceId}/groups/{groupId}/...`)
4. IF an API call returns a 4xx error with a structured error body, THEN THE API_Client function SHALL propagate the error so that UI components can extract and display the specific error message

### Requirement 12: Loading and Error States

**User Story:** As a user, I want clear feedback when data is loading or when errors occur, so that I understand the current state of the interface.

#### Acceptance Criteria

1. WHILE data is being fetched for any self-service tab, THE system SHALL display a loading skeleton or spinner consistent with the existing application loading patterns
2. IF a data fetch fails, THEN THE system SHALL display an error message with a "Retry" button that re-triggers the failed request
3. WHILE a mutation (request, cancel, swap action) is in progress, THE system SHALL disable the triggering button and show a loading indicator to prevent double-submission
4. WHEN a mutation succeeds, THE system SHALL update the local UI state optimistically or refetch the affected data to reflect the change immediately
5. IF a mutation fails, THEN THE system SHALL restore the previous UI state and display the error message from the API response

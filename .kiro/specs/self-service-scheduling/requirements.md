# Requirements Document

## Introduction

This feature introduces a **Self-Service Scheduling** mode to the Shifter/Jobuler platform. Currently, the platform supports auto-generated scheduling via the solver service (primarily for closed-base military groups). The new self-service mode enables normal businesses (restaurants, hospitals, security firms) to let their members request shifts from available slots, subject to configurable min/max constraints set by the space or group admin.

The feature introduces a new scheduling mode enum value, a shift request flow with concurrency-safe slot locking, admin-configurable constraints, a shift template system for recurring weekly patterns, and admin manual override capabilities within the self-service context.

## Glossary

- **Scheduling_Mode**: A group-level setting that determines how shifts are filled — either `AutoGenerated` (solver-based) or `SelfService` (request-based)
- **Shift_Slot**: A specific time window on a specific day within a GroupTask that can be claimed by a member. Defined by task, date, start time, and end time
- **Shift_Request**: A member's intent to claim a specific Shift_Slot. Transitions through states: Pending → Approved or Rejected
- **Request_Window**: A configurable time period during which members are allowed to submit Shift_Requests for an upcoming schedule cycle
- **Shift_Template**: A recurring weekly pattern of Shift_Slots that the system generates automatically for each scheduling cycle
- **Slot_Lock**: A short-lived concurrency mechanism that prevents two members from claiming the same Shift_Slot simultaneously
- **Min_Shifts**: The minimum number of shifts a member must hold per scheduling cycle, configured at the group level
- **Max_Shifts**: The maximum number of shifts a member may hold per scheduling cycle, configured at the group level
- **Scheduling_Cycle**: A repeating time period (typically one week) for which shift slots are generated and requests are collected
- **Waitlist**: An ordered queue of members who want a specific Shift_Slot that is currently full
- **Space_Admin**: A user with administrative permissions in the current space (owner or elevated role)
- **Group_Admin**: A user with `PeopleManage` or equivalent permission for a specific group
- **Shift_Request_Service**: The application-layer service responsible for processing shift requests, enforcing constraints, and managing slot availability
- **Slot_Availability_Engine**: The component that reads current slot state, checks capacity, and determines whether a slot can accept a new request

## Requirements

### Requirement 1: Scheduling Mode Configuration

**User Story:** As a space admin, I want to set a group's scheduling mode to either auto-generated or self-service, so that I can choose the appropriate scheduling approach for my organization.

#### Acceptance Criteria

1. THE Group entity SHALL support a `SchedulingMode` property with exactly two values: `AutoGenerated` and `SelfService`
2. WHEN a space admin sets a group's scheduling mode to `SelfService`, THE system SHALL prevent new solver-based schedule runs from being triggered for that group
3. WHEN a space admin sets a group's scheduling mode to `AutoGenerated`, THE system SHALL prevent new self-service shift requests from being created for that group
4. THE system SHALL default new groups to `AutoGenerated` scheduling mode
5. IF a space admin attempts to change scheduling mode while shift requests with status `Pending` or `Approved` exist for the current scheduling period, THEN THE system SHALL reject the change and return an error indicating that unresolved requests must be cancelled or completed before switching modes
6. IF a solver run is in progress when the scheduling mode is changed to `SelfService`, THEN THE system SHALL allow the in-progress run to complete but SHALL prevent any subsequent runs from being triggered

### Requirement 2: Shift Template Definition

**User Story:** As a group admin, I want to define recurring weekly shift templates, so that the same shift slots are automatically generated each scheduling cycle without manual re-creation.

#### Acceptance Criteria

1. THE system SHALL allow group admins to create Shift_Templates specifying: day of week (Monday through Sunday), start time, end time, required headcount (1 to 999), and associated GroupTask
2. WHEN a new Scheduling_Cycle begins, THE system SHALL generate Shift_Slots from all non-deleted Shift_Templates for that group
3. THE system SHALL allow group admins to create, update, and delete Shift_Templates without modifying already-generated Shift_Slots that have at least one approved Shift_Request
4. WHEN a Shift_Template is updated, THE system SHALL apply changes only to Shift_Slots in future Scheduling_Cycles that have zero approved Shift_Requests, preserving all Shift_Slots that have at least one approved Shift_Request
5. THE system SHALL validate that Shift_Template start time is before end time and that required headcount is between 1 and 999 inclusive
6. IF a group admin submits a Shift_Template that fails validation, THEN THE system SHALL reject the request and return an error response indicating which fields failed validation and why

### Requirement 3: Shift Slot Generation

**User Story:** As a system operator, I want shift slots to be generated automatically from templates at the start of each cycle, so that members always have upcoming slots available to request.

#### Acceptance Criteria

1. WHEN a Scheduling_Cycle begins for a self-service group, THE system SHALL generate one Shift_Slot per active Shift_Template for each date within the cycle that matches the template's configured day of week
2. THE system SHALL generate each Shift_Slot with status `Open`, zero current assignments, the template's start time, end time, required headcount as capacity, and the associated GroupTask
3. IF the generation process runs more than once for the same Scheduling_Cycle, THEN THE system SHALL skip any Shift_Slot that matches an existing slot on the same group, date, start time, end time, and source Shift_Template (idempotent generation)
4. THE system SHALL associate each generated Shift_Slot with its source Shift_Template identifier and the Scheduling_Cycle identifier for traceability
5. IF no active Shift_Templates exist for a group when generation is triggered, THEN THE system SHALL generate zero Shift_Slots and log a warning
6. IF a Shift_Template references a GroupTask that no longer exists or is inactive, THEN THE system SHALL skip that template during generation and log a warning identifying the affected template

### Requirement 4: Shift Request Submission

**User Story:** As a group member, I want to request a specific shift slot, so that I can choose when I work based on my availability.

#### Acceptance Criteria

1. WHEN a member submits a Shift_Request for an open Shift_Slot, THE Shift_Request_Service SHALL validate that the slot has available capacity (current assignments fewer than the slot's required headcount)
2. WHEN the Shift_Slot has available capacity, THE Shift_Request_Service SHALL approve the request, assign the member to that slot, set the Shift_Request status to `Approved`, and increment the slot's current assignment count
3. WHEN the Shift_Slot is at full capacity, THE Shift_Request_Service SHALL reject the request and return a message indicating the slot is full, along with up to 5 alternative available slots for the same day; if no alternatives exist for that day, the response SHALL include an empty alternatives list
4. WHEN a Shift_Request is being processed, THE Shift_Request_Service SHALL acquire an exclusive Slot_Lock on the target Shift_Slot before reading capacity and release it after the assignment is persisted or the request is rejected
5. WHEN a member already holds the maximum number of shifts (Max_Shifts) for the current Scheduling_Cycle, THE Shift_Request_Service SHALL reject the request and return an error indicating the maximum has been reached
6. IF a member submits a Shift_Request for a Shift_Slot to which they are already assigned or have a pending request, THEN THE Shift_Request_Service SHALL reject the request and return an error indicating a duplicate request
7. IF a member submits a Shift_Request for a Shift_Slot that does not exist or is not in `Open` status, THEN THE Shift_Request_Service SHALL reject the request and return an error indicating the slot is unavailable

### Requirement 5: Min/Max Shift Constraints

**User Story:** As a group admin, I want to configure minimum and maximum shift counts per member per cycle, so that workload is distributed fairly across the team.

#### Acceptance Criteria

1. THE system SHALL allow group admins to configure Min_Shifts and Max_Shifts integer values at the group level, where Min_Shifts accepts values from 0 to 100 and Max_Shifts accepts values from 1 to 100
2. IF a group admin submits a Min_Shifts value greater than Max_Shifts, THEN THE system SHALL reject the configuration and return an error message indicating that Min_Shifts must be less than or equal to Max_Shifts
3. IF a group admin submits a Min_Shifts value below 0 or a Max_Shifts value below 1 or above 100, THEN THE system SHALL reject the configuration and return an error message indicating the valid range
4. WHEN a member has fewer approved shifts than Min_Shifts at the end of a Request_Window, THE system SHALL mark the member as under-scheduled in the cycle summary and send a notification to the group admin listing each under-scheduled member and their current shift count
5. WHEN a member attempts to request a shift that would cause their total approved-plus-pending shift count for the current Scheduling_Cycle to exceed Max_Shifts, THE Shift_Request_Service SHALL reject the request and return an error message indicating the maximum shift limit has been reached
6. THE system SHALL default Min_Shifts to 0 and Max_Shifts to 7 for new self-service groups
7. IF a group admin updates Max_Shifts to a value lower than a member's current approved shift count for the active Scheduling_Cycle, THEN THE system SHALL accept the configuration change but not revoke existing approved shifts

### Requirement 6: Request Window Management

**User Story:** As a group admin, I want to define when shift requests open and close for each cycle, so that I can control the scheduling timeline and ensure fair access.

#### Acceptance Criteria

1. THE system SHALL allow group admins to configure a Request_Window with an open date/time and close date/time expressed as offsets (in whole hours, ranging from 1 hour to 30 days) before the Scheduling_Cycle start
2. THE system SHALL validate that the Request_Window open date/time is earlier than the close date/time and that the close date/time is no later than the Scheduling_Cycle start
3. WHEN the current time is before the Request_Window opens, THE Shift_Request_Service SHALL reject shift requests with a message indicating the date/time when requests will open
4. WHEN the current time is after the Request_Window closes, THE Shift_Request_Service SHALL reject shift requests with a message indicating the window has closed
5. THE system SHALL default the Request_Window to open 7 days before the cycle start and close 1 day before the cycle start
6. IF a group admin updates the Request_Window while it is currently open, THEN THE system SHALL apply the new close date/time immediately and reject any subsequent requests that fall outside the updated window
7. WHEN the Request_Window closes, THE system SHALL notify the group admin of any members who have fewer approved Shift_Requests than Min_Shifts for that cycle

### Requirement 7: Slot Availability Query

**User Story:** As a group member, I want to view all available shift slots for an upcoming cycle, so that I can choose shifts that fit my schedule.

#### Acceptance Criteria

1. WHEN a member queries available slots, THE Slot_Availability_Engine SHALL return all Shift_Slots for the requested cycle that have remaining capacity (total capacity minus current fill count is greater than zero), sorted by date ascending then start time ascending
2. THE Slot_Availability_Engine SHALL include for each slot: date, start time, end time, task name, current fill count, and total capacity
3. THE Slot_Availability_Engine SHALL exclude slots where the requesting member is already assigned
4. THE Slot_Availability_Engine SHALL exclude slots that conflict with the member's existing approved shifts, where a conflict is defined as any overlap in time windows with exclusive endpoints (a shift ending at time T does not conflict with a shift starting at time T)
5. WHILE the Request_Window is closed, THE Slot_Availability_Engine SHALL return the slot list with a boolean flag indicating read-only status and a message field indicating that requests are not currently accepted, and the system SHALL reject any request attempts made against the returned slots
6. IF the requested Scheduling_Cycle does not exist or has no generated Shift_Slots, THEN THE Slot_Availability_Engine SHALL return an empty list with no error

### Requirement 8: Shift Request Cancellation

**User Story:** As a group member, I want to cancel a previously approved shift request, so that I can free up the slot if my availability changes.

#### Acceptance Criteria

1. WHEN a member cancels an approved Shift_Request, THE system SHALL increment the slot's available capacity by 1 and set the request status to `Cancelled`
2. IF a member attempts to cancel a Shift_Request and the Request_Window is closed and the current time is past the configured cutoff time (default: 24 hours before shift start), THEN THE system SHALL reject the cancellation and return an error message indicating the cancellation window has passed
3. IF a Waitlist exists for the cancelled slot, THEN THE system SHALL automatically assign the slot to the next member on the waitlist and notify that member of the assignment
4. IF the cancellation would bring the member's scheduled shift count below Min_Shifts for the period, THEN THE system SHALL allow the cancellation and set the member's scheduling status to `Under_Scheduled`
5. WHEN a member submits a cancellation, THE system SHALL require a cancellation reason (between 1 and 500 characters) and record the cancellation timestamp, actor, and reason in the audit trail
6. IF a member attempts to cancel a Shift_Request that is not in `Approved` status, THEN THE system SHALL reject the request and return an error message indicating only approved requests may be cancelled

### Requirement 9: Waitlist for Full Slots

**User Story:** As a group member, I want to join a waitlist for a full slot, so that I am automatically offered the slot if someone cancels.

#### Acceptance Criteria

1. WHEN a member requests a full Shift_Slot, THE system SHALL offer the option to join the Waitlist for that slot
2. THE Waitlist SHALL maintain order based on request timestamp (first-come, first-served)
3. WHEN a slot becomes available due to cancellation, THE system SHALL notify the first waitlisted member and hold the slot for a configurable acceptance period, defaulting to 60 minutes and configurable between 15 minutes and 24 hours at the group level
4. IF the waitlisted member does not accept within the acceptance period, THEN THE system SHALL remove that member's waitlist entry and offer the slot to the next member on the waitlist. IF no members remain on the waitlist, THEN THE system SHALL return the Shift_Slot to `Open` status with available capacity
5. WHEN a waitlisted member accepts the offer, THE Shift_Request_Service SHALL process the request as a standard shift request (subject to Max_Shifts validation). IF Max_Shifts validation fails at acceptance time, THEN THE system SHALL remove the member from the waitlist, notify the member that their limit has been reached, and offer the slot to the next waitlisted member
6. THE system SHALL allow members to remove themselves from a Waitlist at any time. IF the member has an active slot offer at the time of removal, THEN THE system SHALL treat the removal as a decline and offer the slot to the next waitlisted member
7. THE system SHALL prevent a member from joining the same Waitlist more than once per Shift_Slot

### Requirement 10: Admin Manual Override

**User Story:** As a group admin, I want to manually assign or unassign members to shift slots, so that I can handle exceptions and resolve scheduling conflicts.

#### Acceptance Criteria

1. THE system SHALL allow group admins to assign a member to any Shift_Slot regardless of capacity limits or Max_Shifts constraints
2. THE system SHALL allow group admins to remove a member from a Shift_Slot regardless of Min_Shifts constraints
3. WHEN an admin assigns a member to a slot, THE system SHALL create an approved Shift_Request with an `admin_override` flag and record the admin's identity in the audit trail
4. WHEN an admin removes a member from a slot, THE system SHALL cancel the existing Shift_Request with reason `admin_removed` and process the Waitlist for that slot if the slot's current assignment count drops below its required headcount
5. IF the requesting user does not hold the `SchedulePublish` permission, THEN THE system SHALL reject the override request with an error indicating insufficient permissions
6. IF the admin attempts to assign a member who is already assigned to the target Shift_Slot, THEN THE system SHALL reject the request with an error indicating the member is already assigned
7. IF the admin attempts to remove a member who is not currently assigned to the target Shift_Slot, THEN THE system SHALL reject the request with an error indicating no active assignment exists
8. IF the specified member is not a member of the group that owns the target Shift_Slot, THEN THE system SHALL reject the request with an error indicating the member does not belong to the group

### Requirement 11: Concurrency and Slot Locking

**User Story:** As a developer, I want the shift request process to be concurrency-safe, so that two members cannot claim the last available spot simultaneously.

#### Acceptance Criteria

1. WHEN a Shift_Request is being processed, THE Shift_Request_Service SHALL acquire an exclusive Slot_Lock on the target Shift_Slot before reading capacity
2. THE Slot_Lock SHALL use PostgreSQL advisory locks scoped to the Shift_Slot identifier
3. IF the Slot_Lock cannot be acquired within 5 seconds, THEN THE Shift_Request_Service SHALL return a retry-later response to the requesting member that includes an error message indicating the slot is temporarily unavailable
4. THE Slot_Lock SHALL be released within the same database transaction in which the assignment is persisted or the request is rejected
5. THE system SHALL NOT hold Slot_Locks across multiple database transactions
6. WHEN multiple concurrent Shift_Requests target the same Shift_Slot and only one capacity unit remains, THE Shift_Request_Service SHALL accept exactly one request and reject all others with a response indicating the slot is at full capacity
7. IF the Shift_Request_Service determines that current_assigned_count equals max_capacity for the locked Shift_Slot, THEN THE Shift_Request_Service SHALL reject the Shift_Request and release the Slot_Lock within the same transaction

### Requirement 12: Shift Swap Between Members

**User Story:** As a group member, I want to propose swapping one of my shifts with another member's shift, so that we can accommodate each other's schedule changes without admin intervention.

#### Acceptance Criteria

1. WHEN a member proposes a swap, THE system SHALL create a Swap_Request with status `Pending`, specifying the initiator's assignment and the target member's assignment, provided both assignments belong to the same published schedule version and both shifts start in the future
2. WHEN a Swap_Request is created, THE system SHALL notify the target member of the swap proposal within the same request
3. WHEN the target member accepts the swap, THE system SHALL validate that neither member would violate a time-overlap constraint (as determined by the existing ConflictDetector) in the resulting assignment state, and if valid, atomically reassign both shifts (swap initiator gets target's slot, target gets initiator's slot) and mark the Swap_Request as `Accepted`
4. IF either member would violate a time-overlap or rest-period constraint after the swap, THEN THE system SHALL reject the acceptance, keep both assignments unchanged, and return an error message indicating which member has the conflict and which shifts overlap
5. WHEN the target member declines the swap, THE system SHALL mark the Swap_Request as `Declined` and notify the initiator
6. IF the initiator cancels a Swap_Request that is still in `Pending` status, THEN THE system SHALL mark it as `Cancelled` and no notification is sent to the target member
7. IF a Swap_Request remains in `Pending` status for more than 72 hours, THEN THE system SHALL automatically mark it as `Expired` and notify the initiator
8. IF a member proposes a swap on a shift that already has a `Pending` Swap_Request, THEN THE system SHALL reject the proposal with an error message indicating that the shift already has a pending swap request
9. IF the initiator does not own the offered assignment or the target member does not own the requested assignment, THEN THE system SHALL reject the swap proposal with an error message indicating invalid ownership

### Requirement 13: Notifications for Self-Service Events

**User Story:** As a group member, I want to receive notifications about shift request outcomes and waitlist updates, so that I stay informed about my schedule.

#### Acceptance Criteria

1. WHEN a Shift_Request is approved, THE system SHALL send an in-app notification and a push notification to the requesting member confirming the shift name, date, and time of the assignment
2. WHEN a Shift_Request is rejected due to full capacity, THE system SHALL send a notification to the requesting member indicating the rejection reason and including a link to the available shifts list for the same date range
3. WHEN a waitlisted member is offered a slot, THE system SHALL send a notification including the shift name, date, time, and an acceptance deadline of 24 hours from the time the notification is sent
4. WHEN the Request_Window opens for a new cycle, THE system SHALL send a notification to all group members within 5 minutes of the window opening, indicating the cycle date range and the window closing date and time
5. WHEN a swap proposal is received, THE system SHALL send a notification to the target member including the proposing member's name, the shift being offered, and the shift being requested in return
6. WHEN a member's total assigned shifts at Request_Window close is below the minimum shift count configured for the group, THE system SHALL send a notification to both the member and the group admin indicating the member's name and the shortfall count
7. IF notification delivery via push fails, THEN THE system SHALL retain the notification in the in-app notification list so the member can view it upon next login
